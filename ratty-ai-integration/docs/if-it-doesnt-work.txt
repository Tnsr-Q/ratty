Those commands are not ordinary terminal output. The OSC payload is intercepted before rendering and converted into direct scene mutations.

## The correct three-part integration

### 1. OSC 777 interception

The CLI emits something like:

```text
ESC ] 777 ; ratty:git.diff ; visualize=true BEL
```

The VT callback recognizes OSC `777`, reconstructs the payload, parses it into `RattyAiCommand`, and sends it into the channel.

```rust
fn unhandled_osc(
    &mut self,
    _screen: &mut vt100::Screen,
    params: &[&[u8]],
) {
    let [b"777", namespace, payload @ ..] = params else {
        return;
    };

    let Ok(namespace) = std::str::from_utf8(namespace) else {
        return;
    };

    if !namespace.starts_with("ratty:") {
        return;
    }

    let payload = payload
        .iter()
        .filter_map(|part| std::str::from_utf8(part).ok())
        .collect::<Vec<_>>()
        .join(";");

    let sequence = if payload.is_empty() {
        namespace.to_owned()
    } else {
        format!("{namespace};{payload}")
    };

    if let Some(command) = parse_osc_777(&sequence) {
        let _ = self.ai_command_tx.try_send(command);
    }
}
```

### 2. Channel into Bevy ECS

The parser may run as part of terminal processing, while Bevy owns scene state. The channel is the isolation boundary.

```rust
use bevy::prelude::*;
use crossbeam_channel::{Receiver, Sender, unbounded};

#[derive(Resource)]
pub struct RattyAiReceiver(pub Receiver<RattyAiCommand>);

pub type RattyAiSender = Sender<RattyAiCommand>;

pub fn setup_ai_bridge(app: &mut App) -> RattyAiSender {
    let (tx, rx) = unbounded();

    app.insert_resource(RattyAiReceiver(rx));

    tx
}

pub fn bridge_system(
    receiver: Res<RattyAiReceiver>,
    mut writer: MessageWriter<RattyAiCommand>,
) {
    for command in receiver.0.try_iter() {
        writer.write(command);
    }
}
```

### 3. Plugin registration and live handlers

```rust
pub struct RattyAiPlugin;

impl Plugin for RattyAiPlugin {
    fn build(&self, app: &mut App) {
        app.add_message::<RattyAiCommand>()
            .init_resource::<AiState>()
            .init_resource::<BookmarkRegistry>()
            .init_resource::<MacroRecorder>()
            .init_resource::<ReactiveTriggers>()
            .add_systems(
                Update,
                (
                    bridge_system,
                    handle_mode,
                    handle_warp,
                    handle_objects,
                    handle_flash,
                    handle_cursor,
                    handle_ps,
                    handle_fs,
                    handle_git,
                    handle_net,
                    handle_ai_state,
                    handle_panes,
                    handle_history,
                    handle_collab,
                    handle_sound,
                    handle_avatar,
                    handle_macro,
                    handle_reactive,
                ),
            );
    }
}
```

Then Ratty startup becomes:

```rust
let mut app = App::new();

let ai_tx = setup_ai_bridge(&mut app);

let runtime = TerminalRuntime::spawn(
    &app_config,
    &RuntimeOptions {
        command: cli.command.clone(),
        working_dir: Some(std::env::current_dir()?),
    },
    ai_tx,
)?;

app.insert_resource(runtime)
    .add_plugins(DefaultPlugins)
    .add_plugins(TerminalPlugin)
    .add_plugins(RattyAiPlugin)
    .run();
```

## The important architectural interpretation

Your command categories form a **real-time terminal control protocol**:

| Layer            | Responsibility                               |
| ---------------- | -------------------------------------------- |
| `ratty-ai`       | AI-facing control client                     |
| OSC 777          | Transport through the active terminal stream |
| `osc_parser.rs`  | Protocol decoder                             |
| Channel          | Thread/runtime boundary                      |
| `RattyAiCommand` | Typed command contract                       |
| `RattyAiPlugin`  | Bevy registration                            |
| Handler systems  | Live scene execution                         |

The floating terminal and GitHub Pages automation demonstrate Ratty’s rendering capability. This experiment adds something materially different:

> **An AI can perceive a task, execute shell operations, and continuously manipulate the terminal’s spatial presentation as part of its behavior.**

That supports sequences such as:

```text
AI begins reasoning
→ terminal pulses
→ repository tree expands spatially
→ changed files illuminate
→ test process appears as an orb
→ failing process flashes red
→ AI repairs code
→ success sound plays
→ confidence aura increases
→ result is bookmarked
```
